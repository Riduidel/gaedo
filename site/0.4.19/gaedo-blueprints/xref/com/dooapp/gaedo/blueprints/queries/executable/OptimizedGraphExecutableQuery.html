15  /**
16   * Optimized executable query.
17   * How is it optimized ? Instead of simply searching through all vertices linked to class vertex (which always exist) through
18   * Object.classes edge, this one tries to find the vertex linked to the smallest solution space. How ?
19   * As an example, suppose one want to find, using our test beans, elements matching that query :
20   * <pre> 
21   *  AND
22   *	  Posts.note ==? 4.0
23   *	  Posts.author.login ==? "user login"
24   *	  Object.classes contains Post.class
25   * </pre>
26   * This class will start by extracting query roots (vertices to which solutions of this query MUST be linked) by calling {@link #getPossibleRootsOf(VertexTest)}
27   * In this example, this method call will return
28   * <pre> 
29   * 4.0:long => { Property[note] }
30   * "user login":string => { Property[author], Property[login] }
31   * Post:class => { Property[class] }
32   * </pre>
33   * One question remain : from these three vertices, which one will be linked (through the given property path) to the smallest number of vertices ?
34   * To answer this question, the {@link #findBestRootIn(Map)} method creates a {@link VertexValueRange} object which can back-navigate the edges representing these properties
35   * and directly give us the best query root.
36   * 
37   * @author ndx
38   * 
39   */
